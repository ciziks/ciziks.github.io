Validando Patches com checkpatch antes do Envio no kw send-patch

Implementação da issue #1163: integração do checkpatch.pl ao fluxo do kw send-patch, garantindo que cada patch seja validado quanto ao estilo do kernel Linux antes de ser enviado via git send-email.

1) O Problema · Issue #1163

O kw send-patch envia patches diretamente via git send-email sem nenhuma validação prévia de estilo. A documentação do kernel Linux recomenda explicitamente que todo desenvolvedor rode o checkpatch.pl antes de submeter qualquer patch às listas de discussão.

Na prática, o fluxo manual exigia uma etapa extra antes de cada envio: 

# fluxo atual: o desenvolvedor precisa lembrar de rodar isso manualmente
kw codestyle --check meu-patch.patch   # ou ./scripts/checkpatch.pl diretamente
kw send-patch --send -1

Eu mesmo enfrentei essa fricção no ciclo do kernel linux. Durante o desenvolvimento, toda vez que ia enviar um patch eu rodava o checkpatch para garantir que a formatação estava correta. O processo sendo manual tornava fácil esquecer a etapa e enviar patches com problemas de estilo. A ideia da issue é que essa validação aconteça automaticamente como parte do fluxo do kw send-patch.

2) Infraestrutura Existente

Antes de implementar, mapeei o que o kw já oferecia para montar a solução sem duplicar lógica:

  • src/codestyle.sh: já sabe como invocar o checkpatch.pl e encapsula as opções de configuração;
  • pre_generate_patches() em src/send_patch.sh: já gera os patches em ${KW_CACHE_DIR}/patches antes do envio via git format-patch;
  • find_kernel_root(): já é chamado em mail_send() para localizar a raiz do kernel, onde o checkpatch.pl reside em scripts/checkpatch.pl.

Com esses três elementos já implementados no projeto, a solução seria encaixar uma nova função entre pre_generate_patches() e a construção do comando git send-email.

3) A Solução

As mudanças necessárias foram divididas em quatro partes:

3.1) Nova opção em etc/send_patch.config

Para que o usuário possa desabilitar a validação sem usar flags, adicionei uma opção no arquivo de configuração do send-patch: 

# Run checkpatch on each patch before sending. Set to 'no' to disable.
checkpatch_before_send=yes

3.2) Nova função run_checkpatch_on_patches()

A função itera sobre todos os patches gerados no cache e executa o checkpatch.pl em cada um. Uma decisão de design importante: ela não aborta no primeiro erro. Em vez disso, percorre todos os patches, acumula as falhas e retorna 1 somente ao final, caso ao menos um tenha falhado. Assim o desenvolvedor vê os problemas de todos os patches antes de ser notificado do abort.

3.3) Integração em mail_send()

Logo após pre_generate_patches() e antes de construir o comando git send-email, a chamada foi encaixada com três guards combinados: 

if [[ -n "$kernel_root" && "${send_patch_config[checkpatch_before_send]}" != 'no' &&
    -z "${options_values['NO_CHECKPATCH']}" ]]; then
  run_checkpatch_on_patches "$kernel_root" "$flag"
  if [[ "$?" != 0 ]]; then
    complain 'Patches failed checkpatch, aborting send.'
    return 1
  fi
fi

Os três guards garantem que o checkpatch só executa quando: (1) estamos dentro de uma kernel tree (-n "$kernel_root"); (2) a opção não foi desabilitada no config (checkpatch_before_send != 'no'); e (3) o usuário não passou --no-checkpatch (-z "${options_values['NO_CHECKPATCH']}").

3.4) Flag --no-checkpatch no parser

Para usuários que queiram pular a validação pontualmente sem alterar o arquivo de configuração, adicionei a flag --no-checkpatch ao parser de argumentos do comando.

4) Implementação

4.1) src/send_patch.sh

Adicionei a função run_checkpatch_on_patches(kernel_root, flag), que localiza o scripts/checkpatch.pl a partir da raiz do kernel. Se o script não for encontrado (por exemplo, fora de uma árvore do kernel), a função retorna silenciosamente sem erro, pois o usuário não deve ser bloqueado por isso: 

function run_checkpatch_on_patches()
{
  local kernel_root="$1"
  local flag="$2"
  local checkpatch_options="${send_patch_config['checkpatch_opts']}"
  local checkpatch
  local patch_cache="${KW_CACHE_DIR}/patches"
  local ret=0

  checkpatch="$(join_path "$kernel_root" 'scripts/checkpatch.pl')"
  if [[ ! -f "$checkpatch" ]]; then
    warning 'checkpatch.pl not found, skipping style check.'
    return 0
  fi

  for patch_path in "${patch_cache}/"*; do
    if ! is_a_patch "$patch_path"; then
      continue
    fi
    say "Running checkpatch on: $(basename "$patch_path")"
    local cmd="perl $checkpatch"
    [[ -n "$checkpatch_options" ]] && cmd+=" $checkpatch_options"
    cmd_manager "$flag" "$cmd $patch_path"
    if [[ "$?" != 0 ]]; then
      ret=1
    fi
  done

  return "$ret"
}

Dois detalhes importantes na implementação: quando o checkpatch.pl não é encontrado, a função emite um warning (não silêncio) e segue adiante, útil para desenvolvedores fora de uma kernel tree. O checkpatch é invocado via cmd_manager com perl, o que respeita o flag de modo de execução (real vs. simulado) e suporta opções extras via checkpatch_opts no config.

Também adicionei load_kworkflow_config ao final do script para garantir que as configurações (incluindo checkpatch_opts) sejam carregadas corretamente.

5) Testes

Os testes foram escritos em tests/unit/send_patch_test.sh usando o shunit2. Cobri dois grupos de cenários:

5.1) test_run_checkpatch_on_patches

Cinco cenários para a função isolada:

  • Ausência do checkpatch.pl = deve retornar 0 (skip silencioso), sem bloquear o usuário.
  • Nenhum patch no cache = sem arquivos .patch, retorna 0.
  • Um patch válido = checkpatch.pl retorna 0, função retorna 0.
  • Falha em um único patch = checkpatch.pl retorna 1, função retorna 1.
  • Múltiplos patches com falha = verifica que ambos os patches são processados antes do retorno, não apenas o primeiro.

5.2) test_mail_send_checkpatch

Teste de integração ponta a ponta a partir de um kernel root simulado. Cobre três casos:

  • Abort quando o checkpatch falha em algum patch;
  • Bypass via --no-checkpatch: o envio prossegue mesmo com falhas;
  • Bypass via checkpatch_before_send=no no arquivo de configuração.

6) Fluxo Resultante

Com a feature implementada, o kw send-patch --send -1 agora segue o fluxo: 

kw send-patch --send -1
  → pre_generate_patches()       # gera patches em ${KW_CACHE_DIR}/patches
  → run_checkpatch_on_patches()  # valida cada patch com checkpatch.pl
      ✗ erro → aborta e exibe o output do checkpatch para cada patch com falha
      ✓ ok   → continua
  → git send-email ...           # envia os patches

Para pular a validação quando necessário, o usuário tem duas opções equivalentes: 

# via flag pontual
kw send-patch --send --no-checkpatch -1

# via configuração persistente em ~/.config/kw/send_patch.config
checkpatch_before_send=no

A escolha de executar as validações por padrão e oferecer o escape hatch via --no-checkpatch é intencional: o objetivo é que o caminho seguro seja o caminho fácil. A opção de validar todos os commits de um multi-patch mas abortar somente se ao menos um falhar é uma escolha de design que pode ser revisitada no futuro.

8.1) Erro na CI - sh-checker (shfmt)

Após abrir o PR, os testes automatizados de CI acusaram falha no sh-checker:

O shfmt reportou um problema de indentação em src/send_patch.sh: um espaço extra na continuação do if dentro de mail_send(): 

--- src/send_patch.sh.orig
+++ src/send_patch.sh
@@ -166,7 +166,7 @@
   if [[ -n "$kernel_root" && "${send_patch_config[checkpatch_before_send]}" != 'no' &&
-      -z "${options_values['NO_CHECKPATCH']}" ]]; then
+    -z "${options_values['NO_CHECKPATCH']}" ]]; then

A correção foi remover o espaço extra na linha de continuação do if para alinhar com o padrão esperado pelo shfmt -i=2, e atualizar o commit com git commit --amend. Após isso, a CI passou com sucesso.
