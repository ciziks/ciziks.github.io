1) O Projeto kworkflow

O projeto de contribuição em software livre que eu e minha dupla Felipe Ribeiro de Souza escolhemos, após sugestão e discussão com o professor Paulo e o monitor David, foi colaborar com o kworkflow (kw).

O kw é uma ferramenta de linha de comando escrita em Bash que simplifica o workflow de desenvolvimento do kernel Linux: build, deploy em VMs, envio de patches por e-mail, gerenciamento de configurações, entre outros. É mantido principalmente por Rodrigo Siqueira e colaboradores da comunidade brasileira de desenvolvimento do kernel, o que facilita a comunicação e o suporte durante a contribuição.

A escolha fez sentido por dois motivos principais: durante a primeira etapa da disciplina usamos intensamente o kw para compilar e enviar nosso patch para o subsistema IIO, então já tínhamos familiaridade com o projeto; e por ser escrito em Bash, o código é acessível sem precisar de um ambiente de compilação complexo.

Primeiramente, cogitamos contribuir atualizando os textos de ajuda (-h) de alguns comandos, pois durante a etapa do kernel percebemos uma defasagem entre a documentação oficial e o que era exibido no terminal. O Felipe também notou que alguns workflows do GitHub Actions estavam usando versões depreciadas de actions. Entretanto, após avaliar as issues e conversar com o David, optamos por seguir um caminho mais robusto: implementar features e corrigir bugs solicitados diretamente pelos mantenedores.

3) Configurando o Ambiente

O primeiro passo prático foi criar um fork do projeto no GitHub e clonar localmente. O monitor David nos orientou a trabalhar na branch unstable, que é a branch de desenvolvimento ativo do kw, e avisá-lo quando o PR fosse criado, pois ele mesmo faria a revisão. 

# Criando o fork pelo GitHub e clonando
git clone git@github.com:ciziks/kworkflow.git
cd kworkflow
git remote add upstream git@github.com:kworkflow/kworkflow.git

# Criando a branch de trabalho a partir da unstable
git fetch upstream
git checkout -b unstable upstream/unstable

Com o ambiente configurado, estávamos prontos para começar a trabalhar nas issues. A primeira escolhida foi a #1257, aparentemente um bug no kw send-patch com a flag de versão, por ser bem delimitada e ter uma reprodução clara descrita pela própria Melissa.
